# コーディング原則

言語・フレームワークに依存しない普遍的なコーディング原則。
コードベース全体の実装判断およびレビュー基準として適用する。

---

## 関数設計

1関数に1つの責務を持たせ、複数の責務を混在させない。

- 主語（何を）と述語（何をする）が明確に言える単位を目安に分割する（例: 「ユーザーリストをフィルタリングする」「認証トークンを検証して結果を返す」）
- 関数の処理内容と返り値が明確にわかる名前をつける
- 可能な限り純粋関数を優先し、副作用はシステム境界（ロガー・DI・Context 等）に限定する
- 深いネストを避けるため、ガード節（early return）を優先する
- パラメーターが多い場合はオブジェクトにまとめるか、責務を分割する
- なぜその実装にしたか（why）が自明でない場合にコメントを追加する

## イミュータビリティ（重要）

原則として新しいオブジェクトを生成し、既存オブジェクトの直接変更を避ける。パフォーマンス上明確なボトルネックがある場合、または外部 API 仕様により不可避な場合のみ例外とする。その場合は理由をコメント等で明示する。

```
// 擬似コード
WRONG:  modify(original, field, value) → changes original in-place
CORRECT: update(original, field, value) → returns new copy with change
```

> 注: イミュータブルなデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を実現する。

## データ構造

データの関心事を明確に分離し、意図を名前で表現する。

- 異なる関心事のデータを同じコレクションに混在させない
- 文脈から意味が即座に分からない数値・文字列は定数化する（例: `3600` → `SECONDS_PER_HOUR`）

## ファイル構成

ファイルは小さく保ち、肥大化を避ける。

- 1ファイルは単一の明確な責務のみを持つ
- 大きなファイルは責務単位で分割する

## エラーハンドリング

エラーは必ず明示的に扱う（伝播・変換・ログ・リトライ等）。

- 処理の各段階で適切に変換または伝播する
- 利用者に公開されるエラーは、理解可能な形式で提示する
- 内部処理では詳細なエラーコンテキストを記録する
- エラーをサイレントに飲み込まない

## 入力バリデーション

システム境界（API 入口・UI 入力・外部 I/O）では常にバリデーションを行う。

- 処理前にすべてのユーザー入力をバリデーションする
- 可能な場合はバリデーションルールを明示的に定義して検証する
- 明確なエラーメッセージで即座に失敗させる
- 外部データ（API レスポンス・ユーザー入力・ファイル内容）を信頼しない

## コード品質チェックリスト

作業完了前に確認:
- [ ] 意図が伝わる完全な単語で命名されている（一般的な略語のみ許可: `id`, `idx`, `num`, `err`, `ctx` 等）
- [ ] エラーをサイレントに飲み込まず、処理の各段階で明示的に扱われている
- [ ] ハードコードされた値がない（定数または設定を使用）
- [ ] ミューテーションなし（イミュータブルパターンを使用）
