# アーキテクチャ原則

言語・フレームワークに依存しない普遍的なアーキテクチャ設計原則。
システム構造の変更および新規設計時のレビュー基準として適用する。

---

## Repository パターン

**適用条件:** データの永続化・取得が必要な場合（データベース、外部 API、ファイルなど）。インメモリのみで完結する処理には不要。

データアクセスを一貫したインターフェースの背後にカプセル化する。

- 基本操作（findAll, findById, create, update, delete）を提供する
- ドメインに特化したクエリメソッドを追加してよい（例: findActiveUsersByOrganization）
- Repository は「集約単位」でデータを扱い、テーブル単位ではない
- Repository は永続化責務のみを持ち、業務ロジックを含めない
- トランザクション境界はアプリケーション層で管理し、Repository は制御しない
- 具象実装がストレージの詳細を処理（データベース、API、ファイルなど）
- ビジネスロジックはストレージメカニズムではなく、抽象インターフェースに依存

## API レスポンスフォーマット

**適用条件:** 新規エンドポイントを設計するとき、または既存エンドポイントのレスポンス形式を変更するとき。

### REST API

HTTPステータスコードをステータスの主体とする。

- 成功/失敗は HTTP ステータスコードで表現（200/201/400/404/500 等）
- 単純な CRUD API ではエンベロープを使用しない
- ただし横断的メタデータ（traceId, links 等）が必要な場合は統一エンベロープを導入してもよい
- コレクション取得時はページング情報を含める（total, page, limit）

### RPC（tRPC / Connect / gRPC-Web 等）

アプリケーションレベルの成功/失敗は型で表現する。HTTP ステータスはトランスポート層として使用し、業務エラーの一次判定には用いない。

- 成功: `{ "data": T }` — `error` フィールドは含めない
- エラー: `{ "error": ErrorT }` — `data` フィールドは含めない
- ErrorT は少なくとも code（安定した識別子）と message を含む構造を持つ
- 上記を `Data | Error` の Union Type として排他的に定義し、型による分岐を強制する
- アプリケーションエラーをHTTPステータスコードに自動マッピングするフレームワーク（Connect, tRPC等）を優先し、「常に200を返す」純粋なJSON-RPCは避ける

> **原則**: REST = HTTP規約に従う / RPC = 型（Schema）規約に従う

## Star Topology パターン

**適用条件:** 同一責務カテゴリ（Service、Store、Handler 等）が3つ以上になるとき。2ファイル以下の小規模構成では任意。

同一責務カテゴリ内のモジュールは互いに直接依存しない。各モジュールはコア（コンポジションルート・ホスト・共有インターフェース）を通じてのみ通信する。

- ルート（Express Router、React Router の Route 等）は他のルートをインポートしない
- 同一カテゴリ（Service同士、Store同士など）の直接依存は禁止する
- 上位レイヤーが下位レイヤーを参照することは許可する
- ツール（MCP tool、カスタム Hook 等）は他のツールをインポートしない
- すべての結合はコンポジションルートで依存性注入によって行う
- 依存関係は有向非循環グラフ（DAG）を維持する

> 注: 上記により依存グラフがハブ・アンド・スポーク構造を維持し、N² の結合や連鎖的な破壊を防ぐ。

**クロスカテゴリ依存の解決策:**
- 共通処理を下位の共有レイヤー（Shared / Util）に切り出す（Shared は横断的関心 pure function・汎用ユーティリティ に限定する）
- 上位のユースケース層（Facade / UseCase）で複数モジュールを組み合わせる

**適用対象:** 単一プロセスアプリケーション。マイクロサービスやイベント駆動アーキテクチャでは、メッセージキューやイベントバス等の別パターンを使用すること。

**Escape Hatch:** このパターンから外れる必要がある場合は、理由を ADR（アーキテクチャ決定記録）として残し、技術的負債として明示的に管理する。
